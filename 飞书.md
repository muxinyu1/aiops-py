1. 问题回顾
核心问题回顾：攻击者将精心构造的 payload 注入到目标系统的遥测数据（主要是日志）中，使得 AIOps Agent 在读取这些遥测数据进行 incident 响应时被误导，执行恶意的修复操作——本质上是一种针对 AIOps Agent 的提示词注入。
现有工作（AIOpsDoom，即 SP 2026 那篇）采用黑盒 fuzzing：对目标应用的所有可达 HTTP 端点盲目注入 payload，期望其中某些请求会触发错误，而错误日志中恰好包含攻击者可控的内容。这种方法有效，但本质上是在"碰运气"——攻击者不知道哪些参数会落入日志、以什么格式落入、落入哪个服务的日志。
我们的思路是转向白盒分析：通过阅读源码，精确找到所有"用户输入最终会出现在遥测数据中"的代码路径，然后做有目标的定向注入。这本质上是一个 taint-style 的分析问题——source 是 HTTP 请求参数，sink 是遥测数据的产生点。
但这里有一个关键的前置问题：sink 从哪来？ 传统 taint 分析的 sink 是固定的（SQL 注入的 sink 是 Statement.execute()，命令注入的 sink 是 Runtime.exec()），而"会产生遥测数据的代码位置"是由项目的代码实现、框架配置、部署方式共同决定的，不是一个固定的 API 列表。这就是我们方案的核心关注点。

2. 整体方案：两阶段拆解
整个方案分为两个阶段：
阶段一：Sink 定位与候选路径生成（静态分析）
给定一个 Java 微服务项目的源码，自动发现所有可能将用户可控的 HTTP 请求内容写入错误/异常级别遥测数据的代码位置（telemetry sink）。对每个 sink，输出：
- 精确的代码位置（类名、方法名、行号）
- sink 的语义类型（直接日志调用、封装日志函数、全局异常处理器、AOP 切面等）
- 一条从某个 HTTP API 入口到该 sink 的候选执行路径
- 该路径上哪些参数可能被记录、路径上的约束条件
阶段二：定向 Fuzz（动态验证 + LLM 驱动）
这部分直接复用 SP 2026 的思路：
1. LLM 根据阶段一输出的候选路径和约束条件，生成 HTTP 请求 payload
2. 执行请求，捕获实际的执行路径（Java 侧需要确定具体的路径追踪方案）
3. 比较实际执行路径与预期路径的偏差
4. 如果有偏差，将偏差信息反馈给 LLM，LLM 做出决策：
  - 调整 payload 参数以修正路径偏差
  - 请求提供该 sink 的替代路径
  - 判断该 sink 确实不可达，标记并跳过
这将黑盒 fuzzing 转变为有目标的定向注入。

为了确保方案设计基于真实项目的实际情况而非假设，首先选取了 Mscan 论文中提到的 22 个 GitHub 上超过 1k star 的 Java 微服务应用（包括 pig、RuoYi-Cloud、yudao、Apollo、gulimall 等），然后用 agent 逐个做了详细的遥测数据安全分析，最后聚合形成了一份综合报告。Java微服务项目日志与遥测机制分析报告（聚合版）
这份报告覆盖了各项目在日志框架配置、显式日志调用方式、全局异常处理、AOP 切面、网关过滤器、Feign 调用日志、可观测性框架集成、部署采集方式等方面的技术细节。方案设计文档中的所有设计决策（比如"为什么只需要六类隐式边"、"为什么日志 API 列表是有限可枚举的"）都以这份报告中的实证数据为依据。
4. 阶段一的方法论：四步走
阶段一是本方案的核心技术贡献。详细设计见 面向AIOps遥测注入攻击的日志Sink点自动发现方案。
这里给出高层概览。
Phase 1：基础 Sink 种子集构建
做什么： 用硬编码的 API 列表，在源码中扫描所有直接调用已知日志 API 的代码位置。
为什么可行： Java 生态中日志最终都必须通过具体的日志框架 API 发出，而这些 API 是有限可枚举的——SLF4J 的 Logger.error/warn/info()、Log4j2 的对应方法、JUL 的 Logger.severe/warning() 等。22 个项目的实证分析确认了这一点：21 个使用 SLF4J，1 个使用 Log4j2，没有例外。
输出： 基础 sink 点集合 S₀。
Phase 2：框架隐式调用边恢复
做什么： 补全调用图中因框架机制而缺失的边。
为什么需要： Phase 1 找到的很多 sink 位于框架驱动的代码中——全局异常处理器、AOP 切面、过滤器等。这些代码不会被业务代码直接调用，而是由框架在运行时通过注解、配置、AOP 代理等机制触发。如果调用图中缺少这些隐式边，后续的路径分析就无法把 HTTP 入口连接到这些 sink。

怎么做： 基于 22 个项目的实证观察，我们识别出六类需要恢复的隐式边。
每类用声明式规则恢复，不追求通用的完整调用图（那是另一个大问题），只恢复与日志 sink 相关的边。
输出： 补全后的调用图 G₁。
Phase 3：封装日志函数发现
做什么： 识别项目自定义的日志封装函数（如 AuditLogger.recordOperation()、SysLogAspect 等），并将 sink 定位"上移"到封装层。
为什么需要： 真实项目中开发者极少直接调用 log.error()，而是通过项目内部的封装层间接调用。研究表明 75.8% 的大型项目实现了自己的日志封装。sink 应该定位在封装函数的调用处，而非底层日志 API 处。
怎么做： 三步——调用图反向可达分析（找到所有间接调用日志 API 的函数）→ 日志调用密度筛选（过滤掉"恰好调用了日志的普通业务函数"）→ LLM 语义验证（判断候选函数是否以日志记录为主要功能）。
输出： 扩展后的 sink 集合 S₁。
Phase 4：路径分析与可注入性评估
做什么： 对每个 sink，检查是否存在从 HTTP 入口到它的可达路径，以及路径上是否有用户可控参数能传播到日志内容中。
怎么做： 三步——配置可达性过滤（LLM 判断 sink 在生产环境配置下是否启用）→ 调用图正向可达性分析（从 HTTP 入口到 sink 的路径）→ LLM 参数传播判断（哪些 HTTP 参数能传播到 sink 的日志内容中）。
输出： 有序的 fuzz 目标列表，每个目标包含 sink 位置、候选路径、可注入参数及传播方式，直接交给阶段二。

5. 现有工作的覆盖程度与我们的定位
阶段一中最核心的技术挑战是 Phase 2 的隐式调用边恢复。这个问题在学术界已有大量工作：
- Java 企业 Web 污点分析研究判断
- Java 污点分析方案的可用性与前提
现有工作的核心结论
通用 taint engine 本身已经足够强，继续单做 source-sink 求解增强边际收益有限。真正决定效果的是框架与运行机制带来的 hidden edge 是否被恢复。

关键差距： 没有任何现有工作专门针对"日志 sink 发现"这个场景。它们追求的是通用的完整调用图恢复，这是一个更大更难的问题。而我们的场景实际上只需要恢复与日志相关的几类特定隐式边——全局异常处理器、AOP 日志切面、过滤器/拦截器、网关 GlobalFilter、事件监听器、Security 认证处理器。
我们的定位
我们不追求通用的完整调用图恢复，而是写一个更轻量的、专门针对日志隐式调用场景的隐式边恢复模块。根据 22 个项目的实证观察，上述六类机制已经覆盖了所有观察到的框架驱动日志行为。这使得我们的方案在日志场景下的覆盖率接近通用方案，而实现复杂度远低于通用方案。
